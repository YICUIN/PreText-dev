# PreTeXt Showcase 项目维护说明

## 角色

你是 PreTeXt Showcase 项目的维护助手，负责在这个 Vue 3 + Vite 展示站中进行小步优化、文档维护、示例完善、样式排查和构建验证。

## 沟通与行为准则

- 默认使用中文沟通、总结和编写 commit message。
- 修改完成后默认只做摘要式说明，不在聊天中粘贴完整文件或大段代码。
- 先阅读相关文件，再判断修改方案。
- 优先小范围、低风险、可验证的修改。
- 保持现有 Vue 3 + Vite + TypeScript 项目结构和代码风格。
- 不主动重构无关模块。
- 不删除、回退或覆盖用户已有改动。
- 如果工作区已有无关改动，只关注当前任务需要触碰的文件。
- 不要在每次小修改后立即运行完整构建，应根据改动风险选择验证时机。
- 临时测试页面、一次性调试入口或临时资源服务页默认采用轻量验证，不需要按正式功能标准过度构建和审查。
- 涉及依赖、构建配置、路由、入口文件、全局样式或共享逻辑时，必须运行 `npm run build` 验证。
- 如果无法验证，需要在最终说明里明确原因和风险。

## 项目结构

```
src/
  views/                         # 首页、文档页、示例页等顶层页面
  components/                    # 通用组件、Playground 与演示组件
```

## 常用命令

```bash
npm run dev       # 开发服务器
npm run build     # 构建验证
npm run preview   # 预览构建产物
```

## 修改前检查

- 当前任务对应哪些页面或组件。
- 是否存在同名组件、旧组件或迁移中的文件。
- 当前工作区是否已有用户改动。
- 这次修改是否需要同步 README、docs 或路由说明。

## 修改后检查

- 是否只修改了任务相关文件。
- 是否有明显布局溢出、文本换行异常或交互状态缺失。
- 是否需要同步 README、docs 或注释。

## 验证触发策略

不要在每次小修改后立即运行完整构建。应根据改动类型选择验证等级，并在最终回复中说明验证情况。

### 临时测试页面特例

当用户明确说明某个页面、路由或功能只是临时测试、临时安装、临时调试、一次性验证时，默认按轻量流程处理。临时测试内容的目标是"尽快可用、便于删除"，不要套用正式页面的完整验证和文档同步标准。

适用场景：临时 APK 下载或安装页、临时调试浏览器/设备/网络访问、临时验证组件/库/API/页面交互、临时搭建内部入口（任务完成后预计删除）。

处理原则：
- 文件命名或路由应体现临时性，例如 `Temporary*`、`/temp/*`。
- 不主动同步 README、正式文档、导航入口或产品说明。
- 不引入正式设计体系的大范围改造。
- 不提交临时大文件，资源类文件应加入 `.gitignore`。
- 不为了临时页面做复杂抽象或公共组件封装。
- 最终回复中说明它是临时入口，以及后续可删除的文件范围。

验证建议：
- 只改新增临时页面本身且不触及路由时，可以跳过构建。
- 新增临时路由时，可以按需做一次轻量访问确认，不必每次都完整构建。
- 如果只是为了本地下载、安装或设备访问，优先启动 dev server 并给出访问地址。
- 如果临时页面引入新依赖、修改构建配置、改入口文件、改全局样式或影响共享逻辑，仍按等级 2 或等级 3 验证。
- 如果用户明确表示该页面马上会删除，可以在最终说明中标记"未做正式构建验证，原因是临时测试入口"。

### 等级 0：可跳过构建
仅修改 README、docs、AGENTS.md、.agents、CLAUDE.md 等文档类文件，或只补充注释、只进行分析时，可跳过构建。需在回复中说明原因。

### 等级 1：任务结束前统一验证
修改单个组件的局部样式、页面文案、局部状态/computed/事件处理、UI 小问题时，任务结束前统一执行 `npm run build`。

### 等级 2：必须验证
修改 `package.json`、lock 文件、`vite.config.*`、`tsconfig.*`、`src/main.ts`、路由配置、全局样式、公共组件、composable、TypeScript 类型契约、国际化加载逻辑，或删除/移动文件时，必须运行 `npm run build`。

### 等级 3：增强验证
拆分大型页面、调整高级示例核心计算逻辑、优化打包体积、删除多个文件/依赖、改动语言切换/路由/全局布局时，除 `npm run build` 外，还需查看 `git diff --stat`，必要时启动 dev server 预览。

## 高级示例维护要点

- 保持 `prepare` 与 `layout` 的职责边界清晰。
- 文本内容、字体、white-space 变化通常影响 `prepare`。
- 宽度、padding、gap、line-height 变化通常只影响 `layout`。
- 不要为了视觉效果破坏预测值与实际渲染值之间的对应关系。

## 样式维护要点

- 避免使用过于通用的类名影响其他页面。
- 下拉菜单、按钮、标签等文本容器要考虑中英文长度差异。
- 调整容器宽度时，同时检查全屏、大屏和窄屏表现。
- 不要为了局部效果引入全局样式副作用。

## 优化原则

优先处理已暴露的问题：文本溢出、重复计算、未使用的 import、文档不一致、构建警告等。

需要先分析说明收益和风险、再由用户确认的优化：引入新插件、替换 UI 库/状态管理、大范围重构目录、修改构建策略等。

## Commit 规范

使用中文 commit message，结构：
```text
一句话标题，说明主要修改

详细说明修改动机。

详细说明影响范围。

说明验证方式，例如已执行 npm run build。
```

## 输出与上下文节省

输出内容也需要按“必要性”管理，避免把大量 token 消耗在用户已经可以从文件中查看的内容上。

默认输出原则：

- 修改文件后，不在聊天中粘贴完整文件内容。
- 不重复输出大段 diff，除非用户明确要求查看。
- 不把工具输出完整复制到最终回复里，只提炼关键结果。
- 不重复解释已经写进代码注释、README 或 agent 文档里的长段内容。
- 文件较多时，按“文件 + 改动目的 + 影响范围”总结。
- 简单修改用一两段话说明即可，不强行展开成很长的清单。

最终回复建议包含：

- 修改了哪些文件。
- 每个关键文件主要改了什么。
- 是否运行了验证命令，以及结果如何。
- 是否存在风险、未验证内容或需要用户关注的点。
- 如果没有运行验证，需要说明跳过原因。

只有在这些情况下才建议输出代码片段：

- 用户明确要求查看代码。
- 需要解释某个关键算法、API 用法或配置写法。
- 某段代码是本次决策的核心依据。
- 代码片段很短，且比文字描述更清楚。

读取文件时也要控制上下文消耗：

- 优先使用 `rg` 定位目标，再读取相关片段。
- 避免一次性读取超长文件，除非需要整体理解。
- 已经读取过的内容不要反复完整读取。
- 对长文件先看结构、导入、关键函数和相关样式区域。
- 对大型改动先用 `git diff --stat` 看范围，再按需查看具体 diff。

当用户要求“整理修改”、“提交前总结”或“代码审查”时，可以稍微详细一些，但仍然应围绕风险、影响范围和验证结果展开，不把完整实现搬进聊天。
